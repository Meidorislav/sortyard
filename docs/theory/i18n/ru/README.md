# Теория: Имитационное моделирование сортировочного центра

> Подготовительный материал к хакатону 2026

---

## 1. Введение в имитационное моделирование

Имитационное моделирование (Discrete-Event Simulation, DES) — метод изучения поведения сложных систем путём построения их компьютерных моделей и проведения экспериментов. В контексте сортировочного центра DES позволяет воспроизвести движение товаров, тары и потоков, выявить узкие места и оценить эффект от изменений без остановки реального производства.

### 1.1. Ключевые понятия

| Понятие | Определение |
|---|---|
| Событие (Event) | Мгновенное изменение состояния системы: прибытие посылки, завершение сортировки, поломка конвейера |
| Сущность (Entity) | Объект, движущийся по системе: посылка, поддон, курьер |
| Ресурс (Resource) | Ограниченный элемент, занимаемый сущностями: сортировщик, конвейерная лента, ячейка склада |
| Очередь (Queue) | Ожидание сущностью освобождения ресурса |
| Время симуляции | Виртуальное время, ускоренное относительно реального (часы за секунды) |

### 1.2. Подходы к моделированию

| Подход | Описание | Применение в SC |
|---|---|---|
| DES (событийное) | Модель движется от события к событию | Основной метод для сортировочного центра |
| Agent-Based | Автономные агенты с правилами поведения | Моделирование роботов-сортировщиков |
| System Dynamics | Непрерывные потоки и запасы | Макроанализ пропускной способности |
| Monte Carlo | Статистическое семплирование | Анализ вариабельности входных потоков |

### 1.3. Алгоритм DES event loop

Ядро движка — **min-heap очередь событий**, отсортированная по виртуальному времени. Главный цикл:

```
EventQueue: min-heap, упорядочен по (sim_time, sequence_number)

loop:
    if EventQueue пуста → стоп
    event = EventQueue.pop()           // O(log n)
    current_sim_time = event.sim_time
    event.execute(state)               // может добавить новые события в очередь
```

Каждое событие реализует единственный интерфейс:

```go
type Event interface {
    Time() int64        // симуляционное время в наносекундах
    Execute(s *State)   // мутирует состояние, может планировать новые события
}
```

Выполнение **однопоточное** — в каждый момент обрабатывается ровно одно событие, что гарантирует детерминированность. Горутины Go используются только для изоляции цикла симуляции от API-сервера, а не для параллелизма внутри цикла.

Если два события имеют одинаковое `sim_time`, порядок определяется `sequence_number` (автоинкремент в момент планирования события).

## 2. Архитектура сортировочного центра

Сортировочный центр (Sort Center, SC) принимает входящие отправления, классифицирует их по направлениям и отправляет в соответствующие исходящие потоки. Типовой центр обрабатывает от 50 000 до 500 000 посылок в смену.

### 2.1. Типовые зоны и процессы

| Зона | Функция | Ключевые параметры |
|---|---|---|
| Зона приёмки (Inbound) | Разгрузка транспорта, сканирование, взвешивание | Кол-во доков, скорость разгрузки |
| Индукция (Induction) | Поштучная подача на главный конвейер | Производительность (ед./час), % ошибок |
| Главный конвейер (Sorter) | Транспортировка и считывание ШК/QR | Скорость ленты (м/с), длина петли |
| Сортировка (Chutes) | Распределение по направлениям | Кол-во лотков, ёмкость лотка |
| Упаковка (Packing) | Консолидация, упаковка в тару | Нормы времени на упаковку |
| Отгрузка (Outbound) | Погрузка в транспорт по маршрутам | Кол-во доков, окна отгрузки |
| Буферный склад | Временное хранение проблемных отправлений | Ёмкость, время пребывания |

### 2.2. Потоки в системе

**Физические потоки**
- Поток посылок: Inbound → Induction → Sorter → Chutes → Outbound
- Поток тары: Outbound → очистка → Packing → Induction
- Поток персонала: зоны рабочих мест, перемещения между зонами
- Поток транспорта: доки приёмки и отгрузки, парковочные зоны

**Информационные потоки**
- WMS → задания на сортировку
- TMS → расписание транспорта, маршруты
- Сканеры → события о прохождении через точки контроля
- Системы весогабаритного контроля → атрибуты отправлений

### 2.3. Моделирование конвейера в DES

Конвейерная лента — **не** стандартная M/M/c очередь: предметы движутся с фиксированной скоростью по пути известной длины, поэтому время выхода детерминировано по времени входа и состоянию ленты.

Два подхода:

| Подход | Как работает | Когда применять |
|---|---|---|
| **Travel-time event** (используем здесь) | Планируем одно `ConveyorExitEvent` в момент `entry_time + loop_length / belt_speed` | Взаимодействие посылок не моделируется |
| Позиционный | Отслеживаем точную позицию каждой посылки | Плотностные замедления, столкновения |

**Реализация через travel-time:**

```
посылка входит на ленту в момент T:
    планируем ConveyorExitEvent в T + (loop_length / belt_speed)

поломка ленты в момент F, ремонт завершён в F + MTTR:
    для каждого ожидающего ConveyorExitEvent:
        exit_time += MTTR
```

**События сканеров:** размещаем N сканеров на фиксированных позициях вдоль петли. Каждый генерирует `ScanEvent` в момент `entry_time + scanner_position / belt_speed`. При нечитаемом ШК вместо `ChuteSortEvent` планируется `ManualHandlingEvent`.

## 3. Математические основы

### 3.1. Теория массового обслуживания

Теория очередей — математический фундамент DES-моделирования. Нотация Кендалла: **A/B/c/K**, где:
- **A** — распределение входящего потока
- **B** — распределение времени обслуживания
- **c** — число серверов (ресурсов)
- **K** — ёмкость системы

Для большинства узлов сортировочного центра применяется модель **M/M/c**:

| Параметр | Формула | Смысл |
|---|---|---|
| Интенсивность нагрузки (ρ) | `ρ = λ / (c · μ)` | λ — поток входа, μ — скорость обслуживания |
| Утилизация (U) | `U = λ / (c · μ)` | Должна быть < 1 для стабильной очереди |
| Среднее число в очереди (Lq) | `Lq = P₀ · (λ/μ)ᶜ · ρ / (c! · (1−ρ)²)` | Среднее число ожидающих |
| Среднее время ожидания (Wq) | `Wq = Lq / λ` | Закон Литтла: L = λ · W |

> **Практически:** если U приближается к 1 (100%) — в этом узле начинает расти очередь. Цель — держать U в диапазоне 70–85%.

### 3.2. Вероятностные распределения

| Процесс | Распределение | Параметры | Обоснование |
|---|---|---|---|
| Прибытие транспорта | Пуассона / Равномерное | λ = среднее/час | Пуассон — случайные события; Равномерное — по расписанию |
| Время разгрузки | Нормальное / Lognormal | μ, σ | Lognormal — для правоскошенных данных |
| Время сортировки | Экспоненциальное | μ = 1/среднее | Классика для M/M/c моделей |
| Размер партии | Негативное биномиальное | n, p | Учитывает избыточную дисперсию |
| Поломки (MTBF) | Экспоненциальное | λ_fail | Пуассоновский процесс отказов |
| Время ремонта (MTTR) | Lognormal / Weibull | μ, σ | Правый хвост = редкий долгий ремонт |

### 3.3. KPI и метрики эффективности

| KPI | Формула / Описание | Целевое значение |
|---|---|---|
| Throughput | Единиц обработано / час | > плановой нормы |
| Cycle Time | t_выход − t_вход (для единицы) | Минимизация |
| Utilization | Время занятости / Доступное время | 70–85% |
| Queue Length (Lq) | Среднее число ожидающих на узле | Минимизация |
| OEE | Availability × Performance × Quality | > 85% |
| Error Rate | Неверно отсортировано / Всего | < 0.1% |
| Dock Utilization | Занятые доки / Всего доков | < 90% |

### 3.4. ГПСЧ и воспроизводимость

Seed инициализирует генератор псевдослучайных чисел (ГПСЧ) с фиксированной последовательностью. Один и тот же seed всегда даёт одинаковый результат — это необходимо для отладки и честного сравнения сценариев (одни и те же случайные входы, разные параметры).

**Главное правило: один ГПСЧ на каждый независимый случайный процесс**, а не один глобальный. Если добавить новый вызов `rand` в любое место — все последующие значения из общего источника сдвинутся, изменив результаты несвязанных частей модели.

```go
// Правильно: изолированные источники
arrivalRng  := rand.New(rand.NewSource(seed + 1))
failureRng  := rand.New(rand.NewSource(seed + 2))
barcodeRng  := rand.New(rand.NewSource(seed + 3))

// Неправильно: глобальный источник — не детерминирован при конкурентном доступе
rand.Seed(seed)
```

**Период прогрева (метод Уэлча):** симуляция начинается с пустой системы, что нереалистично — очереди не находятся в установившемся режиме. Статистика, собранная в этот переходный период, занижает KPI. Стандартное решение: прогнать симуляцию в течение warm-up периода (например, 30 мин симуляционного времени), затем сбросить все накопители KPI. Длина warm-up должна быть достаточной для стабилизации уровня утилизации.

## 4. Узкие места и методы их выявления

### 4.1. Теория ограничений (ToC)

Метод Голдратта: система всегда ограничена одним самым слабым звеном. Алгоритм из 5 шагов:

1. **Identify** — найти ограничение (узкое место)
2. **Exploit** — максимально использовать имеющийся ресурс
3. **Subordinate** — подчинить остальное процессу работы ограничения
4. **Elevate** — расширить ограничение (добавить ресурс, ускорить)
5. **Repeat** — найти новое ограничение и повторить

### 4.2. Типичные узкие места в сортировочном центре

| Узкое место | Симптомы в модели | Метрика обнаружения |
|---|---|---|
| Зона индукции | Длинная очередь перед конвейером | Queue length > 50 ед. |
| Считыватели ШК | Рост % нечитаемых → ручная обработка | Exception rate > 2% |
| Лотки сортировки | Переполнение → остановка сортера | Chute fill > 80% |
| Зона упаковки | Накопление WIP, срыв окон отгрузки | WIP > норматива |
| Доки отгрузки | Транспорт ждёт, нарушение расписания | Dock wait > 15 мин |
| IT-система (WMS) | Задержки ответа → простой оборудования | Response time > 200 мс |

### 4.3. Методы анализа в симуляции

- **Process Mining** — анализ реальных логов для построения As-Is модели
- **Bottleneck Analysis** — сравнение утилизации и длин очередей по всем узлам
- **Sensitivity Analysis** — изменение одного параметра, наблюдение за KPI
- **What-If Analysis** — сравнение сценариев: добавить сортировщика, изменить расписание
- **Warm-Up Period** — исключение переходного периода из статистики (правило Уэлча)

## 5. Автоматизация и повышение эффективности

### 5.1. Технологии автоматизации

| Технология | Описание | Эффект |
|---|---|---|
| Автоматические индукторы | Роботизированная подача на конвейер | Throughput +30–50%, вариабельность ↓ |
| Кросс-бельтовые сортеры | Высокоскоростная сортировка с минибелтами | До 20 000 ед/час |
| AGV / AMR роботы | Автономные транспортные средства | Замена ручного перемещения тары |
| RFID / Vision AI | Идентификация без ШК, контроль повреждений | Exception rate → ~0% |
| Автоматическая упаковка | Машинная упаковка и маркировка | Cycle time упаковки ↓ 60% |
| Predictive Maintenance | ML-предсказание поломок | Unplanned downtime ↓ 40% |

### 5.2. Оптимизационные подходы

**Расписание и маршрутизация**
- Динамическое планирование смен под входной поток
- Оптимизация маршрутов транспорта (Vehicle Routing Problem)
- Балансировка загрузки лотков — динамическое переназначение направлений

**Управление очередями**
- Priority Queuing — приоритетная обработка срочных отправлений
- Batching — группировка для повышения эффективности упаковки
- Pull vs Push — переход к системе вытягивания для снижения WIP

## 6. Верификация и валидация модели

Верификация — подтверждение, что модель работает так, как задумано. Валидация — подтверждение, что модель достаточно точно отражает реальную систему.

### 6.1. Верификация

- Unit-тестирование компонентов (событий, процессов, очередей)
- Трассировка: пошаговый контроль событий в логах
- Дегенеративные тесты: нулевая нагрузка → очереди пусты; бесконечная → полная утилизация
- Тест воспроизводимости: один seed → один результат

### 6.2. Валидация

- **Face Validation** — экспертная оценка специалистами сортировочного центра
- **Historical Validation** — сравнение KPI с историческими данными реального объекта
- **Sensitivity Analysis** — предсказуемая реакция модели на изменение параметров
- **Statistical Tests** — тест Колмогорова–Смирнова, chi-square для сравнения распределений

### 6.3. Статистическая значимость

Для получения надёжных результатов проводят несколько прогонов с разными seed'ами (10–30 репликаций). Доверительный интервал строится по t-распределению Стьюдента.
