| Атрибут | Значение |
|---|---|
| Версия | 1.0 |
| Дата | Июнь 2026 |
| Тип | Хакатон |
| Стек | Go · PostgreSQL · TypeScript · React |

---

## 1. Обзор продукта

### 1.1. Цель

Разработать интерактивную имитационную модель сортировочного центра, которая визуализирует движение товаров, тары и основных потоков в реальном времени, выявляет узкие места и демонстрирует эффект от оптимизационных решений.

### 1.2. Проблема

Реальные сортировочные центры не могут тестировать изменения конфигурации без остановки производства. Отсутствие инструмента визуальной симуляции приводит к субоптимальным решениям, принятым без достаточной аналитической базы.

### 1.3. Ценностное предложение

- Снижение рисков: тестирование изменений без воздействия на реальный процесс
- Выявление узких мест до их появления в производстве
- Сравнение сценариев оптимизации с количественными метриками
- Образовательная ценность: наглядное понимание работы сортировочного центра

### 1.4. Критерии успеха на хакатоне

1. Работающая симуляция с визуализацией потоков в реальном времени
2. Возможность изменить параметры и увидеть эффект на KPI
3. Выявление и подсветка минимум 2–3 типов узких мест
4. Демо работает стабильно при 5-минутном прогоне симуляции

## 3. Функциональные требования

### 3.1. Симуляционный движок

**FR-SIM-01: Дискретно-событийная симуляция**
Движок реализует DES с очередью событий, отсортированной по виртуальному времени. Поддерживает ускоренное воспроизведение (1x, 5x, 10x, 60x).

**FR-SIM-02: Моделируемые зоны**

- Зона приёмки — N доков с параметризованным временем разгрузки
- Зона индукции — M точек с настраиваемой производительностью
- Главный конвейер — скорость ленты, длина петли, число сканеров
- Зоны сортировки — K лотков с ёмкостью и направлениями
- Зона упаковки — J рабочих мест, нормы времени
- Зона отгрузки — L доков, расписание вывоза

**FR-SIM-03: Типы сущностей**

- Посылка: ID, направление, вес, габариты, статус, временны́е метки
- Транспортное средство: тип, вместимость, расписание
- Тара (поддон/корзина): ID, местоположение, заполненность

**FR-SIM-04: Генератор нагрузки**

Параметры: распределение входящего потока (Пуассона / Равномерное), интенсивность (ед./час), паттерн нагрузки (плоский / пиковый / двухпиковый), доля крупногабаритных, доля нечитаемых ШК.

**FR-SIM-05: Случайные события**

- Поломки оборудования: задаётся MTBF и MTTR для каждого ресурса
- Нечитаемые ШК: % отправлений идут в ручную обработку
- Ошибки сортировки: % отправлений попадают в неверный лоток

| Метод | Путь | Описание |
|---|---|---|
| POST | /api/scenarios | Создать сценарий |

| POST | /api/simulations | Запустить симуляцию |

| POST | /api/simulations/:id/stop | Остановить |

**FR-API-02: WebSocket Stream**

`ws://host/ws/simulations/:id` — стриминг событий в JSON. Клиент подписывается на типы: `entity_move`, `queue_update`, `kpi_update`, `bottleneck_detected`.

### 3.3. Аналитика и выявление узких мест

**FR-ANALYTICS-01: Автоматическое обнаружение узких мест**

- Queue Length Alert: длина очереди > порогового значения (настраивается)
- Utilization Alert: утилизация > 90% на протяжении > 5 мин симуляционного времени
- Chute Overflow: лоток заполнен > 80%, конвейер тормозит
- Dock Congestion: все доки заняты, транспорт ожидает

**FR-ANALYTICS-02: KPI Dashboard (реальное время)**

Throughput (ед./час) · Average Cycle Time · Utilization по зонам · Queue Length · Error Rate · OEE конвейера

**FR-ANALYTICS-03: Сравнение сценариев**

Запуск 2+ сценариев и сравнение итоговых KPI в таблице и на графике.

**FR-UI-01: Главный экран**
- Топологическая карта центра с живыми иконками сущностей
- Цветовая индикация состояния зон (зелёный / жёлтый / красный по загрузке)

**FR-UI-03: Настройка сценария**

- Пресеты: «Базовый», «Пиковая нагрузка», «Минимальный персонал»
- Сохранение / загрузка сценария

**FR-UI-04: KPI панель**

## 4. Нефункциональные требования

| Категория | Требование | Целевое значение |
|---|---|---|
| Производительность | Симуляция 1 часа работы центра | < 10 сек реального времени (60x) |
| Производительность | WebSocket задержка | < 100 мс |
| Производительность | API response time (p95) | < 200 мс |
| Масштаб | Число сущностей одновременно | До 10 000 посылок |
| Надёжность | Детерминированность при одном seed | 100% воспроизводимость |
| Надёжность | Стабильность на 5-мин демо | Обязательно |
| Удобство | Time To First Result | < 30 сек от нажатия Start |
| Код | Покрытие тестами движка | > 70% (unit-тесты) |

## 5. Технический стек и архитектура

### 5.1. Стек

| Слой | Технология | Роль | Обоснование |
|---|---|---|---|
| Симуляция | Go 1.22+ | DES движок, горутины | Скорость, конкурентность |
| API | Go + chi | REST + WebSocket | Низкая задержка |
| БД | PostgreSQL 16 | Сценарии, результаты | JSONB + аналитика |
| Query | sqlc / pgx | Типобезопасный SQL | Без оверхеда ORM |
| Фронтенд | TypeScript + React 18 | SPA | Строгая типизация |
| Графики | Recharts + D3.js | KPI, карта центра | React-native, SVG |
| State | Zustand + React Query | Глобальный стейт + данные | Лёгкость |
| Сборка | Vite | Dev server + build | Скорость HMR |
| Окружение | Docker + docker-compose | Локальный запуск | Простота демо |

### 5.2. Архитектура взаимодействия

```
React SPA ──HTTPS REST──► Go API ──in-process──► Simulation Engine
React SPA ──WebSocket────► Go WS Server                   │
                                                           ▼
                                                    PostgreSQL 16
```

### 5.3. Схема БД

```sql
-- Конфигурации сценариев
scenarios (
  id          UUID PRIMARY KEY,
  name        TEXT,
  config      JSONB,   -- параметры всех зон
  created_at  TIMESTAMPTZ
)

-- Прогоны симуляции
simulation_runs (
  id                UUID PRIMARY KEY,
  scenario_id       UUID REFERENCES scenarios,
  status            TEXT,        -- pending | running | paused | done | error
  seed              BIGINT,
  speed_multiplier  INT,
  started_at        TIMESTAMPTZ,
  ended_at          TIMESTAMPTZ
)

-- Временные ряды KPI (snapshot каждые N сек симуляционного времени)
run_kpi_snapshots (
  run_id       UUID REFERENCES simulation_runs,
  sim_time     BIGINT,          -- секунды симуляционного времени
  throughput   INT,
  avg_cycle_time FLOAT,
  utilization  JSONB,           -- { "inbound": 0.72, "sorter": 0.91, ... }
  queue_lengths JSONB
)

-- Зафиксированные узкие места
bottleneck_events (
  id          UUID PRIMARY KEY,
  run_id      UUID REFERENCES simulation_runs,
  sim_time    BIGINT,
  zone        TEXT,
  type        TEXT,             -- queue_overflow | high_utilization | chute_full | dock_wait
  severity    TEXT,             -- warning | critical
  description TEXT
)

-- Трассировка отдельных посылок (опционально, только при debug)
entity_log (
  run_id      UUID,
  entity_id   UUID,
  event_type  TEXT,
  zone        TEXT,
  sim_time    BIGINT
)
```

## 8. Риски и митигация

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Производительность движка | Средняя | Высокое | Профилировать рано; перейти на in-memory без БД при нужде |
| Сложность WS синхронизации | Средняя | Среднее | Начать с polling, перейти к WS позже |
| Время на визуализацию | Высокая | Высокое | MVP: статичная SVG + числа; анимация — P2 |
| Расхождение скоростей симуляции и рендера | Средняя | Среднее | Буферизировать события на клиенте (очередь 500 мс) |
| Warm-up period в статистике | Низкая | Среднее | Добавить параметр warm-up time, исключать из KPI |

## 9. Глоссарий

| Термин | Определение |
|---|---|
| DES | Discrete-Event Simulation — дискретно-событийное моделирование |
| SC | Sort Center — сортировочный центр |
| WMS | Warehouse Management System |
| TMS | Transport Management System |
| OEE | Overall Equipment Effectiveness |
| MTBF | Mean Time Between Failures |
| MTTR | Mean Time To Repair |
| WIP | Work In Progress — посылки в обработке |
| Throughput | Пропускная способность (ед./час) |
| Utilization | Доля времени, когда ресурс занят |
| Cycle Time | Время прохождения посылки через весь центр |
| Chute | Лоток сортировки |
| Induction | Подача посылок на конвейер поштучно |
| Seed | Начальное значение ГПСЧ для воспроизводимости |
